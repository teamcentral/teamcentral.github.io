CI Screenshot Automation ​

This document describes how to make the Playwright screenshot process (npm run screenshots) triggerable in CI, so documentation screenshots stay in sync with the app automatically.

Current State ​

  • Screenshots are captured locally via npm run screenshots (runs scripts/take-screenshots.mjs)
  • The script builds a static server from build/, mocks all API calls, and uses Playwright to capture 53+ routes
  • Output PNGs are committed to docs/public/screenshots/ and served by VitePress
  • The deploy_docs.yml workflow deploys docs on push to main but does not regenerate screenshots

Proposed Workflow: update_screenshots.yml ​

A new GitHub Actions workflow that:

  1. Builds the app
  2. Runs Playwright screenshot capture with mock data
  3. Compares output against committed screenshots
  4. Either commits updated screenshots (auto-update mode) or fails if screenshots are stale (CI check mode)

Workflow File ​

yaml
name: Update Documentation Screenshots

on:
  # Manual trigger with option to auto-commit
  workflow_dispatch:
    inputs:
      auto_commit:
        description: 'Auto-commit updated screenshots'
        type: boolean
        default: false

  # Optional: run on PRs that touch app source (detect stale screenshots)
  # pull_request:
  #   paths:
  #     - 'src/**'
  #     - 'scripts/take-screenshots.mjs'

jobs:
  screenshots:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm
          cache-dependency-path: '**/package-lock.json'

      - name: Install app dependencies
        run: npm ci

      - name: Install Playwright browser
        run: npx playwright install chromium

      - name: Build app
        run: npx vite build

      - name: Capture screenshots
        run: npm run screenshots

      - name: Check for changes
        id: diff
        run: |
          if git diff --quiet docs/public/screenshots/; then
            echo "changed=false" >> $GITHUB_OUTPUT
            echo "✅ Screenshots are up to date"
          else
            echo "changed=true" >> $GITHUB_OUTPUT
            echo "📸 Screenshots have changed:"
            git diff --stat docs/public/screenshots/
          fi

      - name: Upload screenshot diff (for review)
        if: steps.diff.outputs.changed == 'true'
        uses: actions/upload-artifact@v4
        with:
          name: updated-screenshots
          path: docs/public/screenshots/

      - name: Auto-commit updated screenshots
        if: steps.diff.outputs.changed == 'true' && github.event.inputs.auto_commit == 'true'
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add docs/public/screenshots/
          git commit -m "docs: update screenshots from CI [skip ci]"
          git push

Implementation Steps ​

Phase 1: Manual trigger (lowest effort) ​

  1. Add the workflow file above to .github/workflows/update_screenshots.yml
  2. Developers trigger it manually from GitHub Actions UI when screenshots need refreshing
  3. With auto_commit: true, it pushes updated PNGs directly to the branch

Prerequisites:

  • npm run screenshots must work headless on Ubuntu (it already does — Playwright launches with --no-sandbox)
  • The vite build step skips eslint (use npx vite build instead of npm run build)

Phase 2: PR check (detect stale screenshots) ​

Uncomment the pull_request trigger with paths: ['src/**']. When app source changes, the workflow captures fresh screenshots and:

  • If they match committed PNGs → passes
  • If they differ → fails with a diff artifact showing which screenshots changed

This catches cases where a UI change was merged without updating the docs.

Phase 3: Auto-update on merge to main ​

Add a trigger on push to main with auto-commit enabled:

yaml
on:
  push:
    branches: [main]
    paths:
      - 'src/**'

This keeps screenshots perpetually in sync — every merge that touches src/ automatically regenerates and commits updated screenshots, which then triggers the existing deploy_docs.yml to rebuild the VitePress site.

Phase 4: Integrate with deploy_docs.yml ​

Combine screenshot generation into the docs deploy workflow itself, so screenshots are always fresh at deploy time:

yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npx playwright install chromium
      - run: npx vite build
      - run: npm run screenshots
      - run: npm ci --prefix docs
      - run: npm run --prefix docs build
      - uses: actions/upload-pages-artifact@v3
        with:
          path: docs/.vitepress/dist

This is the most robust approach but adds ~2-3 minutes to the docs deploy (Playwright install + app build + 53 screenshot captures).

Considerations ​

  • Build time: Full screenshot run takes ~60-90s locally. With CI overhead (Playwright install, app build), expect ~3-4 minutes total.
  • Flaky screenshots: Time-based data (dashboard charts, timestamps) uses mock data, so screenshots are deterministic. No flakiness risk from real data.
  • Font rendering: Ubuntu renders fonts slightly differently than macOS. Screenshot appearance may shift slightly when moving from local to CI capture. Run once in CI and commit the result to establish the baseline.
  • Playwright version: Pin the Playwright version in package.json to avoid unexpected browser rendering changes across updates.
